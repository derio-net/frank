# Paperclip AI Orchestrator Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deploy Paperclip v0.3.1 (AI agent orchestrator) on Frank via ArgoCD with Bitnami PostgreSQL, LiteLLM integration, and Cilium L2 LoadBalancer exposure at 192.168.55.212.

**Architecture:** Two ArgoCD apps — `paperclip-db` (Bitnami PostgreSQL Helm chart on Longhorn) and `paperclip` (raw manifests: Deployment, ExternalSecrets, ConfigMap, PVC, LoadBalancer Service). Agents route through existing LiteLLM gateway. Web UI on port 3100.

**Tech Stack:** Paperclip v0.3.1 (custom GHCR image), Bitnami PostgreSQL 14.1.10 (OCI Helm chart), ExternalSecrets for Infisical, Cilium L2 LoadBalancer.

**Design doc:** `docs/superpowers/specs/2026-03-14--orch--paperclip-design.md`

### Task 2: Paperclip-db ArgoCD Application CR

Create the ArgoCD Application CR that deploys Bitnami PostgreSQL into `paperclip-system`. Uses the same Docker registry Helm source as `infisical-postgresql`.

**Files:**
- Create: `apps/root/templates/paperclip-db.yaml`

**Reference:** `apps/root/templates/infisical-postgresql.yaml` for OCI Helm chart pattern.

- [ ] **Step 1: Create the Application CR**

Create `apps/root/templates/paperclip-db.yaml`:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: paperclip-db
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: infrastructure
  sources:
    - repoURL: registry-1.docker.io/bitnamicharts
      chart: postgresql
      targetRevision: "14.1.10"
      helm:
        releaseName: paperclip-db
        valueFiles:
          - $values/apps/paperclip-db/values.yaml
    - repoURL: {{ .Values.repoURL }}
      targetRevision: {{ .Values.targetRevision }}
      ref: values
  destination:
    server: {{ .Values.destination.server }}
    namespace: paperclip-system
  syncPolicy:
    automated:
      prune: false
      selfHeal: true
    syncOptions:
      - ServerSideApply=true
      - CreateNamespace=true
      - RespectIgnoreDifferences=true
  ignoreDifferences:
    - group: ""
      kind: Secret
      jsonPointers:
        - /data
```

Notes:
- No sync-wave annotation — defaults to wave 0, which deploys before the `paperclip` app (wave 1).
- `CreateNamespace=true` ensures `paperclip-system` exists before PG deploys (the explicit namespace manifest in the wave-1 app will reconcile labels).
- `ignoreDifferences` + `RespectIgnoreDifferences=true` prevents ArgoCD from detecting drift on the auto-generated password Secret. This is an improvement over the `infisical-postgresql` reference pattern which lacks both.
- `releaseName: paperclip-db` means the auto-generated Secret will be named `paperclip-db-postgresql` (chart convention: `<releaseName>-postgresql`).
- Chart version `14.1.10` matches the existing infisical-postgresql deployment.

### Task 7: ExternalSecret for GHCR image pull secret

Create the ExternalSecret that syncs a GitHub PAT from Infisical and formats it as a Docker registry credential for pulling from `ghcr.io/derio-net/paperclip`.

**Files:**
- Create: `apps/paperclip/manifests/external-secret-ghcr.yaml`

**Reference:** No existing imagePullSecret ExternalSecret in the cluster — this is a new pattern.

- [ ] **Step 1: Create the ExternalSecret**

Create `apps/paperclip/manifests/external-secret-ghcr.yaml`:

```yaml
# Syncs GHCR PAT from Infisical and formats as Docker registry credential.
# Required to pull ghcr.io/derio-net/paperclip (private image).
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: paperclip-ghcr
  namespace: paperclip-system
spec:
  refreshInterval: 5m
  secretStoreRef:
    name: infisical
    kind: ClusterSecretStore
  target:
    name: paperclip-ghcr
    creationPolicy: Owner
    template:
      type: kubernetes.io/dockerconfigjson
      data:
        .dockerconfigjson: |
          {"auths":{"ghcr.io":{"username":"derio-net","password":"{{ .GHCR_PAT }}"}}}
  data:
    - secretKey: GHCR_PAT
      remoteRef:
        key: GHCR_PAT
```

Notes:
- `type: kubernetes.io/dockerconfigjson` is required for imagePullSecrets.
- The username can be any string for GHCR PAT auth; `derio-net` (the org name) is conventional.

### Task 9: Deployment

Create the Paperclip Deployment. References ConfigMap, ExternalSecrets, PVC, Bitnami PG password Secret, and GHCR imagePullSecret.

**Files:**
- Create: `apps/paperclip/manifests/deployment.yaml`

**Reference:** Design spec lines 130-145 for DATABASE_URL construction, resource limits, and health checks.

- [ ] **Step 1: Create the Deployment**

Create `apps/paperclip/manifests/deployment.yaml`:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: paperclip
  namespace: paperclip-system
  labels:
    app.kubernetes.io/name: paperclip
    app.kubernetes.io/component: server
spec:
  replicas: 1
  selector:
    matchLabels:
      app.kubernetes.io/name: paperclip
      app.kubernetes.io/component: server
  template:
    metadata:
      labels:
        app.kubernetes.io/name: paperclip
        app.kubernetes.io/component: server
    spec:
      imagePullSecrets:
        - name: paperclip-ghcr
      containers:
        - name: paperclip
          image: ghcr.io/derio-net/paperclip:v0.3.1
          ports:
            - name: http
              containerPort: 3100
              protocol: TCP
          envFrom:
            - configMapRef:
                name: paperclip-config
            - secretRef:
                name: paperclip-llm-key
            - secretRef:
                name: paperclip-auth
          env:
            - name: PG_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: paperclip-db-postgresql
                  key: password
            - name: DATABASE_URL
              value: "postgres://paperclip:$(PG_PASSWORD)@paperclip-db-postgresql.paperclip-system.svc:5432/paperclip"
          volumeMounts:
            - name: data
              mountPath: /paperclip
          resources:
            requests:
              memory: 256Mi
              cpu: 250m
            limits:
              memory: 1Gi
              cpu: "1"
          readinessProbe:
            httpGet:
              port: http
              path: /
            periodSeconds: 10
          livenessProbe:
            httpGet:
              port: http
              path: /
            initialDelaySeconds: 30
            periodSeconds: 15
      volumes:
        - name: data
          persistentVolumeClaim:
            claimName: paperclip-data
```

Notes:
- `envFrom` loads all keys from ConfigMap + both ExternalSecret-generated Secrets.
- `env` section uses Kubernetes variable expansion: `$(PG_PASSWORD)` is resolved from the `secretKeyRef` above it.
- The Bitnami chart names its Secret `<releaseName>-postgresql`, so with `releaseName: paperclip-db` it becomes `paperclip-db-postgresql`.
- The container runs as `node:node` user (set in the Dockerfile's `USER node` directive).

## Deployment Deviations

### 2026-03-25 — `paperclip-anthropic` secret made optional

**Symptom:** After deploying a new image tag, the old pod persisted. Rolling update was stalled with the new pod in `CreateContainerConfigError`.

**Root cause:** `ANTHROPIC_API_KEY` was never added to Infisical (`frank-cluster` / `prod` / `/`), so the `paperclip-anthropic` ExternalSecret was stuck in `SecretSyncedError`. The `secretRef` in the Deployment had no `optional: true`, so Kubernetes refused to construct the container spec for the new pod.

**Fix:** Added `optional: true` to the `paperclip-anthropic` `secretRef` in `apps/paperclip/manifests/deployment.yaml`. The `claude_local` adapter simply won't have a key when the secret is absent — acceptable since LiteLLM is the primary backend.

**Affected task:** Task 9 (Deployment manifest)
