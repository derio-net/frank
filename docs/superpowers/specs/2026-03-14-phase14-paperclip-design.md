# Phase 14 — Paperclip AI Orchestrator

**Date:** 2026-03-14
**Status:** Design
**Phase:** 14

## Overview

Deploy [Paperclip](https://github.com/paperclipai/paperclip) (v0.3.1) as an experimental AI agent orchestrator alongside Sympozium. Paperclip organizes agents into virtual companies with org charts, budgets, and governance — a higher-level abstraction than Sympozium's K8s-native PersonaPacks.

Goal: evaluate Paperclip as an alternative/complement to Sympozium; decide later which stays.

## ArgoCD Applications

Two Application CRs in `apps/root/templates/`:

| App | Template | Type | Source |
|-----|----------|------|--------|
| `paperclip-db` | `apps/root/templates/paperclip-db.yaml` | Helm chart | `oci://registry-1.docker.io/bitnamicharts/postgresql` |
| `paperclip` | `apps/root/templates/paperclip.yaml` | Raw manifests | `apps/paperclip/manifests/` |

Both deploy to namespace `paperclip-system` with sync policy: `selfHeal: true`, `prune: false`, `ServerSideApply=true`, `RespectIgnoreDifferences=true`.

### `paperclip-db` — Bitnami PostgreSQL Values

File: `apps/paperclip-db/values.yaml`

Key values:
- `auth.username: paperclip`
- `auth.password`: auto-generated by chart (stored in `paperclip-db-postgresql` Secret)
- `auth.database: paperclip` (chart auto-creates this database on init)
- Single primary replica (no read replicas)
- `primary.persistence.storageClass: longhorn`, size `5Gi`
- `primary.resources`: 256Mi memory limit, 250m CPU limit
- `metrics.enabled: true` (Prometheus exporter sidecar)

### `paperclip` — Application Manifests

Directory: `apps/paperclip/manifests/`

| File | Resource | Purpose |
|------|----------|---------|
| `namespace.yaml` | Namespace | `paperclip-system` with baseline PodSecurity |
| `deployment.yaml` | Deployment | Paperclip app, 1 replica, port 3100 |
| `service-lb.yaml` | Service (LoadBalancer) | Cilium L2 at 192.168.55.212:3100 |
| `pvc.yaml` | PersistentVolumeClaim | `/paperclip` data volume, Longhorn 2Gi |
| `configmap.yaml` | ConfigMap | Non-secret env vars |
| `external-secret-llm.yaml` | ExternalSecret | LiteLLM credentials (API key + base URL) |
| `external-secret-auth.yaml` | ExternalSecret | `BETTER_AUTH_SECRET` |
| `external-secret-ghcr.yaml` | ExternalSecret | GHCR image pull secret |

## Container Image

Paperclip does not publish official container images. We build from source and push to GHCR under the `derio-net` org.

- **Registry:** `ghcr.io/derio-net/paperclip`
- **Tag:** `v0.3.1` (matching upstream release)
- **Build:** `docker build -t ghcr.io/derio-net/paperclip:v0.3.1 .` from cloned repo
- **Push:** `docker push ghcr.io/derio-net/paperclip:v0.3.1`
- **Pull auth:** imagePullSecret from ExternalSecret (GHCR PAT with `read:packages` scope)

This is a manual operation for now. Automation (GitHub Actions on fork) can be added if Paperclip becomes permanent.

## LLM Backend Integration

Same pattern as Sympozium: ExternalSecret with `target.template` merges two values:

```yaml
target:
  template:
    data:
      OPENAI_API_KEY: "{{ .llmkey }}"
      OPENAI_BASE_URL: "http://litellm.litellm.svc:4000"
```

Paperclip uses the OpenAI SDK internally, so setting `OPENAI_BASE_URL` routes all model calls through the in-cluster LiteLLM gateway. Available models (via LiteLLM): local models via Ollama, OpenRouter cloud models.

| Infisical Key | K8s Secret | Used By | Contains |
|---------------|------------|---------|----------|
| `PAPERCLIP_LITELLM_KEY` | `paperclip-llm-key` | Deployment `envFrom` | `OPENAI_API_KEY` + `OPENAI_BASE_URL` |
| `PAPERCLIP_AUTH_SECRET` | `paperclip-auth` | Deployment `envFrom` | `BETTER_AUTH_SECRET` |
| `GHCR_PAT` | `paperclip-ghcr` | Deployment `imagePullSecrets` | `.dockerconfigjson` for ghcr.io |

## Deployment Configuration

**ConfigMap env vars:**
```
PORT=3100
HOST=0.0.0.0
SERVE_UI=true
PAPERCLIP_HOME=/paperclip
PAPERCLIP_DEPLOYMENT_MODE=authenticated
PAPERCLIP_DEPLOYMENT_EXPOSURE=private
PAPERCLIP_PUBLIC_URL=http://192.168.55.212:3100
```

**`DATABASE_URL`** — constructed using an init container that reads the Bitnami chart's auto-generated password Secret (`paperclip-db-postgresql`) and writes the full connection string to a shared emptyDir volume. The Deployment then reads `DATABASE_URL` from a file via `envFrom`. Alternatively, since the username and database name are fixed (`paperclip`/`paperclip`), we can use a simpler approach: mount the Bitnami password Secret as an env var (`PG_PASSWORD`) and construct the URL in the ConfigMap as a template. The chosen approach: **use an env var from the Bitnami Secret** (`secretKeyRef` for the password) and hardcode the rest of the connection string in the Deployment spec:

```yaml
env:
  - name: PG_PASSWORD
    valueFrom:
      secretKeyRef:
        name: paperclip-db-postgresql
        key: password
  - name: DATABASE_URL
    value: "postgres://paperclip:$(PG_PASSWORD)@paperclip-db-postgresql.paperclip-system.svc:5432/paperclip"
```

This uses Kubernetes variable expansion (`$(PG_PASSWORD)`) to construct the URL inline.

**Resource requests/limits for Paperclip Deployment:**
- Requests: 256Mi memory, 250m CPU
- Limits: 1Gi memory, 1 CPU
- (Image is ~2GB+ due to bundled Claude Code/Codex/opencode-ai; memory usage at runtime is moderate)

**Health checks:**
- Readiness probe: HTTP GET on port 3100 path `/` (Paperclip serves UI on root)
- Liveness probe: HTTP GET on port 3100 path `/`, `initialDelaySeconds: 30` (Node.js startup + DB migration)

## Gotchas

- **No published Docker images** — must build from source and push to GHCR
- **Paperclip Dockerfile installs Claude Code, Codex, and opencode-ai globally** — the image is large (~2GB+); initial pull will be slow
- **`BETTER_AUTH_SECRET` is required** — Paperclip won't start without it (used for session signing)
- **Port 3100** — Paperclip serves both API and UI on the same port when `SERVE_UI=true`
- **`/paperclip` volume** — stores instance config, agent workspaces, and local state; must persist across restarts
- **DATABASE_URL format** — must be a full PostgreSQL connection string with credentials
- **GHCR pull auth** — cluster nodes need imagePullSecret to pull from private GHCR repos
- **Bitnami PG auto-creates the database** — set `auth.database: paperclip` in values; no manual DB creation needed
- **Cilium L2 IP pool** — verify that 192.168.55.212 is within the configured `CiliumLoadBalancerIPPool` range

## Out of Scope

- Authentik SSO integration (can be added later if Paperclip becomes permanent)
- Agent personas/company configuration (user sets these up via Paperclip UI after deployment)
- High availability (single replica is fine for experiment)
- Automated image builds (manual for now)
- Telegram/Slack bot integration

## Manual Operations

### Build & push container image

```yaml
# manual-operation
id: phase14-build-paperclip-image
phase: 14
app: paperclip
plan: docs/superpowers/plans/2026-03-14-phase14-paperclip-design.md
when: "Before deploying paperclip — image must exist in GHCR"
why_manual: "Upstream repo has no published images; building requires local Docker and GHCR auth"
commands:
  - "git clone https://github.com/paperclipai/paperclip.git /tmp/paperclip && cd /tmp/paperclip && git checkout v0.3.1"
  - "docker build -t ghcr.io/derio-net/paperclip:v0.3.1 ."
  - "echo $GHCR_PAT | docker login ghcr.io -u derio-net --password-stdin"
  - "docker push ghcr.io/derio-net/paperclip:v0.3.1"
verify:
  - "docker pull ghcr.io/derio-net/paperclip:v0.3.1"
status: pending
```

### Create Infisical secrets

```yaml
# manual-operation
id: phase14-create-infisical-secrets
phase: 14
app: paperclip
plan: docs/superpowers/plans/2026-03-14-phase14-paperclip-design.md
when: "Before deploying paperclip — ExternalSecrets need source data"

  - "Create LiteLLM virtual key for Paperclip in LiteLLM admin UI, store as PAPERCLIP_LITELLM_KEY in Infisical"

  - "infisical secrets get PAPERCLIP_LITELLM_KEY"
  - "infisical secrets get PAPERCLIP_AUTH_SECRET"
  - "kubectl get externalsecret -n paperclip-system"

```
