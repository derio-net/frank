## Frank Cluster Gotchas (compact)

One-line reminders only. Each section header points at a per-topic file under `docs/runbooks/frank-gotchas/` for full prose, recovery commands, and dated incident notes ‚Äî agents read those on demand. When adding a new gotcha: one-liner here, full body in the corresponding section file.

### ArgoCD ‚Äî `docs/runbooks/frank-gotchas/argocd.md`
- Notifications: subscribe via `subscribe.<trigger>.<service-name>: ""` (the third dotted segment is the service name, not the type).
- Notifications: native `service.telegram` mis-routes positive chat IDs ‚Äî use `service.webhook.telegram`.
- Notifications-cm is owned by the ArgoCD chart ‚Äî put service/triggers/templates under `notifications.notifiers/.triggers/.templates` in `apps/argocd/values.yaml`; `notifications.secret.create: false`.
- Manual `kubectl patch application ‚Ä¶ --type=merge -p '{"operation":{"sync":...}}'` does NOT inherit `spec.syncPolicy.syncOptions` ‚Äî pass `["ServerSideApply=true","RespectIgnoreDifferences=true"]` explicitly or large CMs blow the 256KB last-applied-config annotation.
- Out-of-bounds symlinks anywhere in the repo lock the entire GitOps loop into `ComparisonError` ‚Äî check after symlink commits: `find . -type l -lname '*../../..*'`.
- Root App-of-Apps re-templates leaf Application specs on every sync; live spec patches (selfHeal off, branch override) are reverted within the sync window. For ops needing scale-to-0 to STICK: suspend **root** selfHeal first, then patch leaf apps `"syncPolicy":{"automated":null}` (leaf `selfHeal:false` alone is insufficient), pause operator owners (awx-operator, vm-operator) too; restore = root selfHeal on + one root sync (re-templates all leaves to git truth). Root does NOT manage itself ‚Äî its live patch needs manual restore.
- The UI LoadBalancer (192.168.55.200) is plain HTTP ‚Äî the Service maps 443‚Üí8080 *plaintext*, so `https://` gets a TLS reset. Use `http://192.168.55.200`.
- App-level health can stick at `Degraded` (stale `lastTransitionTime`, all resources Synced/healthy, `resourceHealthSource: appTree`) after an Application source-path change ‚Äî hard refresh does NOT clear it; the live-state cache needs a controller restart or natural resync. Cosmetic, but masks real health.
- Switching a live Deployment `RollingUpdate`‚Üí`Recreate` via git fails the sync with `spec.strategy.rollingUpdate: Forbidden` ‚Äî the live object keeps an orphan `rollingUpdate:` block; one-time `kubectl patch deployment ‚Ä¶ --type=merge -p '{"spec":{"strategy":{"type":"Recreate","rollingUpdate":null}}}'` then re-sync (same root cause as the Helm variant in storage-secrets-ssa.md).

### Storage / Secrets / SSA ‚Äî `docs/runbooks/frank-gotchas/storage-secrets-ssa.md`
- SOPS-encrypted secrets must NOT be ArgoCD-managed; apply out-of-band from `secrets/`.
- RWO PVC + RollingUpdate deadlocks; use `strategy: Recreate`. Switching strategy via Helm needs a one-time `kubectl patch` to clear the orphan `rollingUpdate:` block.
- ESO: empty `data: []` is rejected; delete the ExternalSecret if all keys are removed.
- ESO `GithubAccessToken` generator (mints GitHub App installation tokens) resolves its `auth.privateKey.secretRef` in the **consuming ExternalSecret's namespace** and IGNORES `secretRef.namespace` (even on a `ClusterGenerator`) ‚Äî the private-key Secret MUST live in the consumer's ns or you get `error getting GH pem from secret: not found`. After moving/fixing the key the ExternalSecret stays `SecretSyncedError` on a cached generatorState until forced: `kubectl -n <ns> annotate externalsecret <name> force-sync=$(date +%s) --overwrite`.
- `envFrom.secretRef` without `optional: true` blocks rolling updates when the Secret is missing.
- AWX operator-managed Postgres on Longhorn CrashLoops with `mkdir '/var/lib/pgsql/data/userdata': Permission denied` ‚Äî the `sclorg/postgresql-15` image runs as UID 26 but the fresh PVC mounts root-owned and the operator emits an empty `securityContext`. Set `postgres_data_volume_init: true` in the AWX CR (root init container chowns the volume; storage-agnostic) rather than relying on `fsGroup`.
- Always `ServerSideApply=true`; always `prune: false`; always `ignoreDifferences` on Secret `/data`.
- Longhorn **v1.11.0 instance-manager leaks anonymous Go heap** (~0.9 GiB/day on busy nodes, unbounded; upstream proxy-connection leak, longhorn#12575, fixed v1.11.1+) ‚Äî when it exhausts a node the kernel reclaim-thrashes instead of OOM-killing: kubelet/CRI wedge silently, node NotReady, NO process dies (took out raspi-1 2026-06-04; **RESOLVED 2026-06-05**: chart bumped to 1.11.2 #467, all old IMs retired, mini-1 3.3‚Üí52 GB / gpu-1 40‚Üí117 GB). Graceful `talosctl reboot` on a wedged node hangs in `stopAllPods` ‚Äî go straight to physical power-cycle (safe: immutable OS, journaled EPHEMERAL).
- Longhorn engine live upgrade (`volume spec.image` patch) moves REPLICAS to the new IM but engines stay in the old IM ‚Äî and **reattaching engines re-JOIN a still-existing old IM** (plain pod restarts retire nothing). Retire old IMs per node: suspend root+leaf Argo automation, scale all old-IM-engine workloads to 0 together, wait NATURAL detach, IM culls at 0 instances, scale back. **Never force-delete `VolumeAttachment`** ‚Äî it yanks mounted ext4 mid-write (JBD2 journal errors, FS shutdown, EIO crashloop; cost ruflo an fsck cycle). Full recipe: `storage-secrets-ssa.md`.
- `talosctl memory` rows start with NODE: AVAILABLE = `$8`, `$7` is CACHE. Replica scheduling disabled on raspi-1/raspi-2 (manual op `stor-longhorn-disable-pi-replica-scheduling`); `layer-1-node-memory-headroom` alert (<1GiB absolute, NOT a ratio) is the early warning. Full incident: `docs/investigations/2026-06-04--stor--raspi-1-memory-wedge-incident.md`.

### Tekton ‚Äî `docs/runbooks/frank-gotchas/tekton.md`
- v1 Task uses `computeResources` not `resources` ‚Äî schema validation silently fails the whole app.
- `$(tasks.status)` returns `"Completed"` (not `"Succeeded"`) when tasks are skipped via `when` ‚Äî accept both.
- PVC workspaces mount as root ‚Äî set `fsGroup` on `taskRunTemplate.podTemplate.securityContext`.
- `runAsUser: 65534` (nobody) ‚Üí `HOME=/` (read-only); set `HOME=/tekton/home` env explicitly.
- Gitea sends `X-Gitea-Event` (not `X-GitHub-Event`) ‚Äî use `cel` interceptor, not `github`.

### Argo Rollouts ‚Äî `docs/runbooks/frank-gotchas/argo-rollouts.md`
- Only `canary` / `blueGreen` strategies ‚Äî stateful + RWO PVC apps stay as plain Deployments.
- `workloadRef.scaleDown` defaults to `never` ‚Äî set `onsuccess` explicitly or both Rollout + chart Deployment serve traffic.
- `workloadRef` scales chart Deployment to 0 ‚Äî add `ignoreDifferences` on `spec.replicas` (`group: apps`, `kind: Deployment`).
- `workloadRef` "leaks" to a healthy-looking Helm Deployment when reconcile aborts pre-workload-phase (missing AnalysisTemplate, missing traffic-router plugin, etc.) ‚Äî the ONLY signal is the controller pod log; `kubectl get rollout` looks identical to a steady-state run.
- Prometheus provider panics on empty result vector ‚Üí metric `phase: Error` ‚Üí 10s-cadence retry ‚Üí canary aborts in ~50s. Verify metric exists + has samples first.
- AnalysisTemplate error queries must catch 4xx not just 5xx (`status!~"2..|3.."`).
- `interval: 1m` only governs Successful/Failed measurements; `Error` retries at 10s (consecutiveErrorLimit default 4 ‚Üí ~40-50s to abort).
- `successCondition`/`failureCondition` use `result[0]` syntax for scalar queries.
- `AnalysisTemplate` has no `inconclusiveCondition` field ‚Äî NaN auto-treats as inconclusive; cap with `inconclusiveLimit`.
- LiteLLM Prometheus is Enterprise-only ‚Äî OSS image emits no `litellm_*` metrics.

### Authentik ‚Äî `docs/runbooks/frank-gotchas/authentik.md`
- Blueprints mount in WORKER pod via `blueprints.configMaps` ‚Äî register new ConfigMaps there.
- Blueprints don't assign providers to embedded outpost ‚Äî manual Django ORM step (see `frank-argocd.md`).
- Embedded outpost needs `AUTHENTIK_HOST` env or forward-auth redirects use `0.0.0.0:9000`.
- API uses Bearer token (not basic auth); 2026.x requires `invalidation_flow` + object-shaped `redirect_uris` + `signing_key` UUID.
- `global.env` applies to both server + worker (avoids duplication).

### Grafana ‚Äî `docs/runbooks/frank-gotchas/grafana.md`
- File-provisioned alerts/dashboards in `apps/grafana-alerting/manifests/` are read-only in UI; edit ConfigMap, push, restart pod (provisioning files are read at boot, not watched).
- 12.x SSE alert rules need 3-step A‚ÜíB‚ÜíC; classic-condition format fails with `sse.parseError`.
- VictoriaLogs alert rules must set `model.queryType: stats` (hits `/select/logsql/stats_query`, returns wide). Default `instant` hits `/select/logsql/query` which returns a long series of log lines, and SSE `reduce` rejects it with `input data must be a wide series but got type long`.
- `kube_pod_status_ready{condition="true"}` false-positives in batch namespaces (Tekton, Jobs); use `kube_deployment_status_replicas_unavailable` instead.
- Provisioning env-var coercion turns numbers into ints ‚Äî use YAML block scalars to force string.
- "Cannot change provenance from 'api' to 'file'" ‚Üí delete the API rows from sqlite first (scale down ‚Üí DELETE ‚Üí scale up).
- Helm chart regenerates admin password Secret on re-render ‚Äî recover with `grafana cli admin reset-admin-password` inside the pod.
- Alertmanager dedup window is 4h after re-provisioning a contact point ‚Äî restart pod to reset.
- Table panels with Prometheus instant queries need `"format": "table"` on targets; use `filterFieldsByName` transform.
- OIDC: secret key must be `GF_AUTH_GENERIC_OAUTH_CLIENT_SECRET` for `envFromSecret`.
- VictoriaMetrics chart `genCA` regenerates webhook caBundle ‚Äî `ignoreDifferences` on the validating webhook caBundle.
- `ALERTS{}` does NOT exist in VM for Grafana-managed alerts ‚Äî use `alertlist` panel type.
- A route `mute_time_intervals` mute leaves the alert `state: active` (only silences/inhibitions flip it to `suppressed`) and Grafana 12's v2 `/alerts` API has NO `mutedBy` field ‚Äî so neither proves a mute worked. Verify a mute by the metric gap: `grafana_alerting_dispatcher_alert_processing_duration_seconds_count` increments (alert dispatched) while `grafana_alerting_notification_latency_seconds_count` stays 0 (nothing delivered), with `grafana_alerting_silences{state="active"}=0` ruling out a silence. The v2 `receivers[]` list reflects ROUTING not delivery; a single entry (no second receiver) confirms `continue:false` stopped evaluation at that route. Used to prove the cert-expiry canary's perma-mute (issue #251).
- The Telegram contact point sends with HTML `parse_mode` ‚Äî alert annotations containing `<‚Ä¶>` (e.g. a `<node-ip>` placeholder), a bare `<`/`>`, or `&` make Telegram reject the message with **400 Bad Request**: the rule fires + dispatches but **silently never delivers** (only signal: `ngalert.notifier` log `failed to send telegram message: webhook response status 400`; `grafana_alerting_notification_errors_total` did NOT capture it). Other receivers on the same alert (e.g. Health Bridge webhook) deliver fine ‚Äî it's Telegram-specific. Keep `<>&` out of `summary`/`runbook` (YAML comments are safe ‚Äî Grafana strips them at provisioning). Caught end-to-end on the 2026-06-08 NIC link-flap rule (`<node-ip>` + `>6`); static validation can't catch it ‚Äî only a real/synthetic firing does.

### Observability digest ‚Äî `docs/runbooks/frank-gotchas/obs-digest.md`
- Falco events reach VictoriaLogs via falcosidekick‚ÜíLoki-push with fields `source`/`priority`/`rule`/`k8s_ns_name` ‚Äî NOT `kubernetes.namespace_name` (that's the fluent-bit collector path for Caddy/CrowdSec). Query Falco with `source:syscall`, never a `kubernetes.namespace_name` filter.
- The daily digest uses a split window: traffic = prior calendar day, security = yesterday 00:00 ‚Üí digest run time, so overnight Critical events aren't reported ~24h late.
- Surge detection counts Hop-edge Caddy requests (bots/scanners/probes included), NOT GoatCounter pageviews (JS-beacon, real browsers only) ‚Äî an edge surge with a flat GoatCounter means automated traffic, not a contradiction.
- Surge baseline is `median(7 hour-of-day samples) or 1`; on a quiet blog the median is 0‚Üíforced to 1, so any trickle reads as a huge ratio. `SURGE_ABS_FLOOR` (default 50, ai-alert-helper Deployment env) is the floor ‚Äî no tier below it. `SURGE_VISITOR_FLOOR` (default 10) gates an URGENT on real GoatCounter pageviews (fail-open if GC unreachable; Major‚Üínon-urgent Notable when no humans). Needs the token `stats` permission (manual-op `obs-goatcounter-token-stats-permission`).
- Frank's own blackbox uptime probe hits the public blog edge (~360/hr via the home egress IP) ‚Äî tagged `Frank-Blackbox-Probe` (blackbox-exporter configmap `http_2xx`/`http_2xx_no_redirect` headers) and excluded by `facts.edge_filter`/`facts.PROBE_UA_TOKEN`. Keep those two in sync (cross-ref comments both ways). Excluding the *vendor default* UA was rejected ‚Äî it would whitelist foreign blackbox exporters / spoofers. After re-tagging, the old-UA hits linger in the 1h window for ~1‚Äì2h (not excluded), transiently inflating `current` until they age out.
- ai-alert-helper image builds via CI: `gh workflow run build-ai-alert-helper.yml [--ref <branch>]` (version-pinned tag, like caddy/openrgb) ‚Äî NOT manual docker. The workflow tag is hardcoded and must be bumped with the version (deferred follow-up: parametrize from pyproject).
- Surge notifications are de-duplicated in-memory (single replica / single uvicorn worker / cron `concurrencyPolicy: Forbid` ‚Üí process-global state is safe; re-arms on restart): edge-triggered (send on a rising tier) + `SURGE_COOLDOWN_HOURS` floor (default 6), else a stateless 15-min cron re-sends the same bot-surge Notable ~4√ó/hour. The surge narrative is evidence-only (cites `top_referrers`/`top_paths`/`top_user_agents`, "undetermined" otherwise) ‚Äî the old prompt pre-seeded "Hacker News" against an empty fact sheet. Watch via the **helper** Deployment logs (`grep "surge (sent|suppressed)"`), not the cron Job (its `curl` drops the body).
- The 0.2.0 Telegram analyst long-polls `getUpdates` ‚Äî Telegram allows ONE consumer per bot token, so the helper Deployment must stay `replicas: 1` + `strategy: Recreate`; messages from non-allowlisted chats are dropped with a WARNING (the gate working, not the bot broken). Verify the LLM loop without Telegram via `POST /ask?dry_run=true`. Playbook = `apps/ai-alert-helper/skill/SKILL.md` (hash-suffixed ConfigMap ‚Üí edits roll the pod; requires the app's `prune: true`). CrowdSec "decisions" in 30d retention are ONLY community-blocklist syncs ‚Äî local scenario triggers have never been observed; an unrecognized line in `crowdsec_activity.other_lines` is news.
- **health-bridge: blindness ‚âÝ death** (v0.4.0+). Grafana's synthetic `DatasourceError`/`NoData` are "monitoring can't see the layer," NOT "the layer died" ‚Äî the bridge now caps them at `degraded` with NO bug (pre-v0.4.0 a datasource outage marked ~10 layers `dead` and filed `[Bug] DatasourceError is dead` issues, all summaries `[no value]`). Heal closes bugs by **feature-ref alone** (`FindOpenBugsByFeature`, alertname-agnostic) so a tracker recovering under its real per-rule name closes a `DatasourceError`-titled bug; `FindOpenBugs` stays alertname-aware only for create-dedup. Stranding still possible when a **fresh Grafana pod eats the `resolved`** (power-outage classic: the new process never fired the alert, so no resolve is sent) ‚Üí tiles stuck, bugs open. Recovery: replay a synthetic `DatasourceError` *resolved* webhook for the stuck `frank-ops#` trackers (idempotent; recipe in `operating/16-health-bridge` + `obs-digest.md`). Incident 2026-06-08.

### Networking ‚Äî `docs/runbooks/frank-gotchas/networking.md`
- Cilium: `lbipam.cilium.io/ips` alone is NOT a sharing directive ‚Äî separate Services need matching `lbipam.cilium.io/sharing-key`. Always check `kubectl get svc -A | grep pending` after deploying a chart that splits one service into multiple Services on a shared LB IP.
- Cilium 1.17 FQDN policies need DNS-proxy initialization on the node first; restart the agent if the LRU isn't ready (stale BPF rules also persist after CNP deletion ‚Äî agent restart clears them).
- MixedProtocolLBService (TCP + UDP on one Service) works on Cilium 1.17 + K8s 1.35 ‚Äî no feature gate needed.
- mosh: flags only in `--ssh="ssh ..."`; `user@host` positional; pin `--server="mosh-server new -p 60000:60015"` to match the Service's port range. Use the per-shell wrappers in `apps/*/client-setup/laptop/`.
- Caddy JSON access logs store the vhost in field `request.host`, NOT in `_msg` (which is literally `"handled request"`). `_msg:"blog.derio.net"` always matches zero ‚Äî filter `request.host:"blog.derio.net"`. Scope edge queries to Hop with `kubernetes.host:hop-1`.
- A flapping physical NIC strips the node's IP off Cilium's direct-routing device ‚Üí agent loops `"IPv4 direct routing device IP not found" / Failed to initialize datapath` ‚Üí ALL pod traffic on that node dies, so every SSH-via-LB session into its pods drops at once (exit 255), while `kubectl exec` still works (rides kubelet, not the Cilium datapath). Diagnose with `talosctl -n <ip> dmesg | grep 'Link is'` (gpu-1 enp3s0/r8169, post-power-outage 2026-06-08); recovery is physical (reseat cable / switch-end port; one chassis port only) ‚Äî Cilium self-heals on stable link, do NOT drain (gpu-1 = only GPU node, agent pods hard-pinned). Reseat is NOT durable (gpu-1 recurred same day); durable fix prepped as Omni ConfigPatch `patches/phase04-gpu/403-gpu1-pcie-aspm.yaml` (`pcie_aspm=off`, reboots gpu-1 ‚Äî operator-applied), then cap to 1G / NIC replace if it doesn't hold. **Originally no alert fired** ‚Äî binary `NotReady`/`Cilium Down` + `for: 5m` rules are blind to "alive but flapping"; now covered by the `layer-1-nic-link-flap` rule `increase(node_network_carrier_changes_total{device=~"en.*|eth.*"}[30m]) > 6` (counter, not the scrape-aliased `node_network_up` gauge). Full incident + monitoring gap: `networking.md`.

### Agent shells ‚Äî `docs/runbooks/frank-gotchas/agent-shells.md`
- `agent-shell-base` parameterizes user via `AGENT_USER`/`AGENT_HOME` (defaults `agent`/`/home/agent`); kali overrides to `claude`/`/home/claude` to preserve PV state. Don't hardcode `/home/claude` in new init scripts.
- s6-overlay v3 in non-root mode needs `S6_KEEP_ENV=1`, `S6_VERBOSITY=2`, `with-contenv` shebangs (`#!/command/with-contenv bash` ‚Äî `/command/`, NOT `/usr/bin/`), and `/run` chown'd to AGENT_UID at image build time.
- `cont-init.d/30-authorized-keys` only fires at pod boot and COPIES (not symlinks) `/etc/ssh-keys/authorized_keys` ‚Äî re-run by hand or restart pod after adding/rotating SOPS-managed keys.
- sshd scrubs container env on login ‚Äî env-dependent commands launched via `ssh agent@host -- cmd` see no `FRANK_C2_*`/`INFISICAL_*`. Use `kubectl exec` or source from `/proc/1/environ`.
- BYOK shells (env-keyed CLIs like hermes) need a numbered `/etc/profile.d/` shim (ConfigMap + `subPath` mount) re-exporting the keys from `/proc/1/environ` into login shells ‚Äî number it BELOW the image's `50-‚Ä¶-motd.sh` so the auth-status check sees the env. Verify with `ssh host 'bash -lc ‚Ä¶'`, never `ssh host -- cmd` (skips profile.d).
- hermes ‚â•0.15 ignores `OPENAI_*` env for chat (provider `auto` ‚Üí openrouter 401) ‚Äî pin via config.yaml `model:` MAPPING (`default:` + `provider: <name>`) + `providers: {<name>: {base_url, key_env}}`; model-string prefixes (`litellm/`, `custom/`, `custom:litellm:`) do NOT pin the provider. config.yaml is PVC state, seeded manually (`orch-hermes-config-provider`).
- `gemma4:12b-64k` + `qwen3.6:35b-a3b-64k` (chart-side `ollama.models.create`, Modelfile num_ctx) are the per-model escape hatch from litellm#12930 ‚Äî server default stays 16384; keep hermes config.yaml `context_length` overrides equal to live reality (all four 64k aliases = 65536, everything else = 16384; manual-ops `orch-hermes-context-budgets` + `orch-hermes-qwen64k-budgets`). Web pages go through `fetch-text <url>` (ConfigMap subPath at `/usr/local/bin`) via the TERMINAL ‚Äî SOUL.md must say "shell command", or small models hallucinate it as a function call. `tool_loop_guardrails.hard_stop_enabled: true` kills degenerate tool loops at ~5 iterations (gemma4 once burned 90 on a `hostname`).
- `shareProcessNamespace: true` is incompatible with s6-overlay v3 (suexec must be PID 1) ‚Äî use shared workspace volume + `kubectl exec -c <other>` for cross-container debugging.
- s6 crashloop bail (5 deaths in 60s) leaves service down silently ‚Äî `s6-svc -u /run/service/<name>` after fixing the root cause.
- tmux-continuum auto-restore only fires when the tmux server starts fresh, not on `tmux source-file ~/.tmux.conf`.
- `/etc/skel/.tmux.conf` only seeds on first boot of a fresh PV ‚Äî existing PVs need a manual `source-file /etc/agent/tmux-resurrect.conf` appended once.
- PVC mounts at `/home/claude` hide all image-baked files under that path ‚Äî entrypoints/configs/templates live outside (e.g., `/opt/`) and seed PVC on first boot.
- secure-agent-kali's `‚öÝ Leftover npm global installation at /usr/bin/claude` warning is intentional ‚Äî both the npm-i-g bootstrap binary and the PV-resident native installer coexist by design.
- Sidecars with `runAsUser` overriding the image default need explicit `HOME` env ‚Äî the binary resolves HOME from `/etc/passwd` of the image-baked user.
- Supercronic auto-reloads on `~/.crontab` change ‚Äî no restart needed.
- `claude install` (and the native build's background auto-updater) buffers its ~245MB download ~17√ó in anon memory via Bun's HTTP client (upstream claude-code#22536, closed not-planned; `BUN_JSC_forceRAMSize` does NOT help ‚Äî it's pre-JSC buffering) ‚Äî peak ~4.2GiB group-OOMs any shell pod with a 4Gi limit, killing sshd and dropping ALL SSH sessions at once (hermes 2026-06-07; hermes now 8Gi). `install.sh` is no safer (its last step re-invokes `claude install`, which re-downloads even when the running binary IS the target version). Memory-safe install: curl the binary to `~/.local/share/claude/versions/<ver>` + symlink `~/.local/bin/claude` (full recipe in `agent-shells.md`).
- vk-local `limits.memory: 4Gi` is too tight in practice ‚Äî `VK_MAX_CONCURRENT_EXECUTIONS=4` does NOT bound the cgroup once the bridge feeds 8+ cards (queued sessions retain memory; new images drift baseline). Keep at 8 Gi until the bridge slot count is bound below the executor cap AND a soak under busy load proves the floor.
- The issue-bridge's 30 s MCP timeout cascades to zombie execution_processes: bridge crash on timeout ‚Üí vk-local request handler future drops ‚Üí `Child::wait()` cancelled ‚Üí setup/cleanup shell scripts exit but never reaped ‚Üí DB rows stuck `status='running'` forever, UI shows workspaces stuck active with no output. Durable fix lives in the bridge code (now `fr_vk.bridge` in `super-fr`).
- secure-agent-kali Dockerfile pins the kali repo to `https://kali.download/kali` (official CDN), NOT the `http.kali.org` redirector ‚Äî the redirector round-robins to community mirrors whose `pool/` lags their `Packages` index during fast rolling transitions (GCC-16 churn, 2026-05), so `dist-upgrade` 404s on debs the index promises (e.g. `gcc-16-base 16-20260322-1`). The build (`agent-images`) is non-deterministic with the redirector.
- The `fr_vk.bridge` daemon (formerly v2 `vk.bridge`) emits its banner + `dry-run complete` to **stderr** (logging), not stdout ‚Äî any smoke test / health check capturing `$(fr-bridge --dry-run)` must use `2>&1` or it asserts on an empty string. This silently broke the `agent-images` kali smoke test on every build after the 2026-05-19 v2-bridge cutover.
- hermes-agent-shell's Hermes venv is **PVC-resident** (`/home/agent/.local/opt/hermes-agent`), seeded on first boot from a `uv --relocatable` image seed at `/opt/hermes-agent` by `cont-init.d/35-hermes-venv-seed` (frank#496) ‚Äî uid-1000-writable, so in-pod patches / `hermes update` PERSIST across restarts. - Agent GitHub auth = a rotating GitHub **App installation token** (ESO-minted, mounted at `/var/run/github/token`), NOT a PAT: git uses the `~/.gitconfig` credential helper, gh uses the `/usr/local/bin/gh` wrapper, both reading the token file. **Always verify auth against a PRIVATE repo** ‚Äî public repos read with no token and MASK auth failures (a public-repo "OK" is a false pass).
- The git credential helper MUST read the token with `$(cat "$f")`, not the bash-only `$(< file)` ‚Äî git runs helpers under `/bin/sh`=dash, where `$(< )` yields an EMPTY password (private-repo 401 "Invalid username or token"; public masks it).
- `gh` needs the wrapper: App tokens rotate (no stable store ‚Üí gh falls back to a stale env value or the revoked `~/.config/gh/hosts.yml` PAT ‚Üí "Bad credentials") and lack **user-context** (`gh auth status` reports "invalid"; repo/issue/PR/GraphQL ops still work). A leftover `gh auth setup-git` host-helper (`credential.https://github.com.helper = !/usr/bin/gh auth git-credential`) silently OVERRIDES the generic helper for github.com and serves gh's stored token ‚Äî `02-credential-migrate` strips it.
- App install is **per-repo + per-org** ‚Äî the token 404s on repos not added to the install; the fr-bridge tracks repos across orgs, so each repo needs coverage (or its own App/token). Full prose + the live-patch stopgap: `docs/runbooks/frank-gotchas/agent-shells.md`.
- The agent-images bump PR body is rendered by `scripts/render_bump_body.py` (called from `agent-images-bump.yml`), which lists the upstream agent-images PR(s) in the `old‚Ä¶new` range with a one-line summary. Enrichment is **best-effort** ‚Äî any `gh api` failure (or a missing pre-bump pin) falls back to the legacy two-line body so the bump PR always opens. The "what changed" filter mirrors agent-images `build.yaml` `paths-ignore: docs/**` (docs-only PRs trigger no rebuild, so they're excluded). ### Process / practice
- **A layer is not "Deployed" until its workflow has been triggered + observed end-to-end.** ArgoCD Synced/Healthy proves artifacts exist; not that they work. Especially load-bearing for canaries, cron, webhook handlers, gated promotions.
- **Tag-driven releases: verify the tag tree contains the new code BEFORE pushing** (`git grep <new-symbol> <tag>` non-empty) ‚Äî health-bridge v0.3.0 was tagged from a stale local main (pre-merge SHA), so GHCR built the OLD code under the new version; nothing in the tag‚Üíbuild‚Üíbump chain catches it (workflow green, ArgoCD Synced, pod Running). Caught only at deploy verification; superseded by v0.3.1 at the merge commit (re-pointing a tag is worse: `IfNotPresent` keeps the stale cached image).
- **Never `fr apply` a path under `archived-plans/`.** It reopens the plan's already-closed phase-issues: fr projects an agentic phase as CLOSED only when it observes a *live merged PR* on the issue, so inline-executed / hand-closed phases can't satisfy the rule retroactively and apply flips their CLOSED issues back to OPEN. `fr apply --all` is safe ‚Äî it walks only `plans/`, never `archived-plans/`. To close a lingering archived-plan issue, hand-close it (`gh issue close`); do NOT reconcile via fr. Durable fix tracked in `super-fr#246`.
- **A just-merged agentic phase's Issue can be *transiently* reopened by `fr apply` ‚Äî even for an active `plans/` entry.** `fr.render._phase_complete` reads the merged-PR signal LIVE via `gh.list_linked_prs` and *ignores* the cached `completion.observed_prs` field; GitHub's cross-reference index lags a merge by a few minutes. An apply firing inside that window sees "no merged PR yet ‚Üí agentic phase incomplete ‚Üí OPEN" and reverts the close (re-adds `fr:ready`, reopens). Trigger is a close (manual `gh issue close`, or a PR `Closes #N` auto-close) colliding with the post-merge propagation window. **This is a race, not the deterministic archived-plan projection bug above** ‚Äî the YAML is fine. **Recovery: re-run `fr apply --yes` once the merge has indexed (a few min later) ‚Äî idempotent, heals the close; a follow-up dry-run apply is then a clean no-op.** Durable mitigation (super-fr#246): have apply trust cached `completion.observed_prs` as a fallback when the live query returns empty (the cache field already exists ‚Äî apply just doesn't read it). Incident: #424 (Phase 3, awx-deployment) reopened 2026-06-01 ~2 min after PR #429 merged; re-applied closed.
