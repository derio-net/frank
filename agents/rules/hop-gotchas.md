## Hop Cluster Gotchas

- Never `source .env` when working on Hop — it overrides KUBECONFIG to Frank. Use `source .env_hop` instead
- Talos control-plane taint must be removed for single-node cluster (`allowSchedulingOnControlPlanes: true` in Talos config)
- PodSecurity namespaces (`caddy-system`, `headscale-system`) must be labeled `pod-security.kubernetes.io/enforce: privileged` for hostPort/privileged pods
- Deployments using `hostPort` (e.g., Caddy on 80/443, Headscale STUN on 3478/UDP) must use `strategy: Recreate` — `RollingUpdate` deadlocks on a single-node cluster because the new pod can't bind ports while the old pod still holds them
- Headplane v0.5+ requires a `config.yaml` ConfigMap — env vars alone are insufficient
- Headscale `extra_records` in DNS config provides split-DNS for mesh-only services — add entries for any new mesh-only service
- `talosctl apply-config --config-patch` patches the base file, not the running config — all patches must be combined in one invocation
- Tailscale DaemonSet must run in kernel mode (`TS_USERSPACE=false`, `privileged: true`) for Caddy to see mesh source IPs
- Headplane v0.5.5 serves at `/admin/` base path (`basename="/admin/"`) — Caddy redirects all non-`/admin*` paths to `/admin/`
- Headplane requires `config_path` pointing to mounted Headscale config with `config_strict: true` — non-strict mode works but logs scary warnings and forfeits upstream support
- Headplane binds IPv4 only — `wget localhost:3000` fails (resolves to `::1`), use `wget 127.0.0.1:3000` to test
- Headplane API key must be injected via `HEADPLANE_HEADSCALE_API_KEY` env var from a Secret
- Blog deployment uses SHA-pinned image tags (`ghcr.io/derio-net/blog:<sha>`) — CI commits the tag update to `deployment.yaml`, ArgoCD syncs automatically. No manual `rollout restart` needed
- Falco: installing a binary at container runtime (`apk add` / `pip install` / `npm i -g`, etc.) writes it to the container's writable upper layer and trips the **Critical** rule "Drop and execute new binary in container" (`EXE_UPPER_LAYER`, MITRE TA0003) — which pages via Telegram. This is a *benign-true-positive*, not a Falco bug: bake the tool into a digest-pinned image (read-only base layer) rather than muting the rule. First hit: headscale-backup CronJob's `apk add sqlite` at 03:00 → switched to `alpine/sqlite` pinned by digest.
- Falco priority routing on Hop: floor is `notice`; only `Critical` reaches Telegram, everything `≥ notice` goes to VictoriaLogs (`192.168.55.225:9428`) via the Loki push protocol. To audit what's actually firing without paging: `priority:<level> | stats by (rule) count()` in LogsQL. "Contact K8S API Server From Container" (Notice, mostly ArgoCD reconcile) is the expected high-volume benign baseline.
- CrowdSec bans nothing if the LAPI's `data`/`config` volumes are emptyDir: every LAPI restart wipes the SQLite DB incl. the agent's registered-machine row, so the agent crashloops with `ent: machine not found` (`cscli machines list` empty), parses zero Caddy logs, and produces zero decisions — ArgoCD still shows Synced/Healthy (silent). Fixed (2026-06-19) by persisting both LAPI volumes onto static hostPath PVs (`type: DirectoryOrCreate`, subdirs of `/var/mnt/hop-data/crowdsec/` on the already-attached Hetzner Volume — no new volume), bound to the chart PVCs (`crowdsec-{db,config}-pvc`) from the PV side via `claimRef` since the chart's PVC template has no `volumeName`. `storageClassName: hetzner-volume` is MANDATORY in values — Hop has no default StorageClass, so an unset class hangs the PVC `Pending` forever. The `claimRef.name` MUST track the chart's `<releaseName>-db-pvc` naming (guarded by `scripts/tests/test_crowdsec_lapi_persistence.py` — a LOCAL guard; no CI runs `scripts/tests/`). Persisting `config` also makes the community-blocklist console enrollment durable across restarts. Two follow-on traps: (1) the agent is a DaemonSet that registers its machine ONLY in an initContainer, so the emptyDir→persistent cutover does NOT self-heal — `kubectl -n crowdsec-system rollout restart daemonset/crowdsec-agent` once to re-register against the fresh DB; (2) `hostPath`+`DirectoryOrCreate` is single-node-safe ONLY — `nodeAffinity: hostname Exists` is a tautology, so a 2nd Hop node would silently get a fresh empty dir and lose the DB; pin to `hop-1` before scaling out.
- CrowdSec STILL bans nothing even with a healthy agent if `container_runtime` is wrong: Talos = **containerd** (CRI logs `<ts> stdout F {json}`), but the chart defaults `container_runtime: docker` → acquisition labels logs `type: docker` → `docker-logs` parser does `JsonExtract(Raw,"log")` which is EMPTY for a CRI line → `caddy-logs`' `UnmarshalJSON` fails → access logs unparsed → zero scenarios → zero bans (ArgoCD green, agent Running). Diagnose with `cscli metrics`: Acquisition shows `Lines read > 0` but `Lines parsed = 0`, and Parser metrics show ONLY `docker-logs`, no `caddy-logs`. Fix (2026-06-19): set top-level `container_runtime: containerd` in `clusters/hop/apps/crowdsec/values.yaml` → `cri-logs` strips the envelope; the acquisition's `program: caddy` label still reaches `caddy-logs`. Note `cri-logs` filter is `type == 'containerd'` (NOT `'cri'`) and does NOT set `evt.Parsed.program` — so the `program: caddy` acquisition label is load-bearing (both halves guarded by `scripts/tests/test_crowdsec_cri_logs.py`). Verify end-to-end with `cscli explain --type containerd --labels program:caddy -f <caddy container log>`.
- CrowdSec STILL goes blind (third silent failure) on **Kubernetes log rotation** even with a healthy agent + correct runtime: it tails `/var/log/containers/caddy-*_caddy-system_*.log`, which are **symlinks** to the kubelet's `/var/log/pods/.../0.log`. The chart hardcodes `force_inotify: true` and defaults `poll_without_inotify: false`, so the per-file tailer follows the resolved inode via inotify and CANNOT follow a kubelet **size-rotation** (`containerLogMaxSize`, renames `0.log`→`0.log.<ts>` WITHOUT a pod restart) — it logs `Re-opening moved/deleted file` then hangs forever in `Waiting for ...log to appear`, parsing zero lines (agent `Running 1/1`, ArgoCD green, zero bans; bit ~19h on 2026-06-20, last ban 09:55Z). The Caddy pod uptime is the tell: pod NOT restarted (e.g. 27d) but acquisition frozen → it was a size-rotation, not pod churn. Diagnose with `cscli metrics`: the source's `Lines read`/`parsed` are non-zero but FROZEN. The agent prints the exact fix at startup (`...is a symlink, but inotify polling is enabled...Consider setting poll_without_inotify to true`). Fix (2026-06-21): add `poll_without_inotify: true` to the `agent.acquisition[]` entry in `clusters/hop/apps/crowdsec/values.yaml` (the ONLY chart-overridable lever — `force_inotify` is hardcoded but orthogonal: it governs directory-watch for NEW files, not the per-file tailer). Polling is stat-based → detects rotation/truncation → re-attaches. Guarded by `scripts/tests/test_crowdsec_log_rotation_poll.py`. Deploying it rolls the DaemonSet, which resumes tailing.
- **CrowdSec ban-pipeline canary** (`clusters/hop/apps/crowdsec-canary/`, 2026-06-21) watches for the silent-failure CLASS above (all three breaks looked green): an in-cluster CronJob (`*/5`) scrapes the agent `:6060/metrics` once/run and compares to the previous run's persisted sample (state PV) — fails if `cs_filesource_hits_total` delta==0 (acquisition hung), or caddy-logs `cs_node_hits_ok_total` delta==0 while reads grow (parse broken), or `/metrics` unreachable (agent crashloop). Pages Telegram **directly** on the **2nd consecutive** fail (dedicated `crowdsec-canary-telegram` Secret in `crowdsec-system` — `falco-telegram` is in `falco-system`, secrets are ns-scoped); emits a `crowdsec-ban-canary verdict=…` heartbeat every run → fluent-bit → VictoriaLogs, where a **Frank Grafana dead-man's-switch** (`crowdsec-canary-heartbeat-stale`, `lt 1` over 20m) pages if the heartbeat itself stops. Stock digest-pinned `python:3-alpine` + script via kustomize `configMapGenerator` (no custom image). The acquisition check ASSUMES the ~360/hr Frank blackbox edge probe as a traffic floor (see the reciprocal note in `obs-digest.md`). Telegram creds are optional → CronJob is healthy (heartbeat-only) before the manual secret is created. Guards: `scripts/tests/test_crowdsec_canary*.py`. Spec: `docs/superpowers/specs/2026-06-21--obs--crowdsec-ban-pipeline-canary-design.md`.
