## Hop Cluster Gotchas

- Never `source .env` when working on Hop â€” it overrides KUBECONFIG to Frank. Use `source .env_hop` instead
- Talos control-plane taint must be removed for single-node cluster (`allowSchedulingOnControlPlanes: true` in Talos config)
- PodSecurity namespaces (`caddy-system`, `headscale-system`) must be labeled `pod-security.kubernetes.io/enforce: privileged` for hostPort/privileged pods
- Deployments using `hostPort` (e.g., Caddy on 80/443, Headscale STUN on 3478/UDP) must use `strategy: Recreate` â€” `RollingUpdate` deadlocks on a single-node cluster because the new pod can't bind ports while the old pod still holds them
- Headplane v0.5+ requires a `config.yaml` ConfigMap â€” env vars alone are insufficient
- Headscale `extra_records` in DNS config provides split-DNS for mesh-only services â€” add entries for any new mesh-only service
- `talosctl apply-config --config-patch` patches the base file, not the running config â€” all patches must be combined in one invocation
- Tailscale DaemonSet must run in kernel mode (`TS_USERSPACE=false`, `privileged: true`) for Caddy to see mesh source IPs
- Headplane v0.5.5 serves at `/admin/` base path (`basename="/admin/"`) â€” Caddy redirects all non-`/admin*` paths to `/admin/`
- Headplane requires `config_path` pointing to mounted Headscale config with `config_strict: true` â€” non-strict mode works but logs scary warnings and forfeits upstream support
- Headplane binds IPv4 only â€” `wget localhost:3000` fails (resolves to `::1`), use `wget 127.0.0.1:3000` to test
- Headplane API key must be injected via `HEADPLANE_HEADSCALE_API_KEY` env var from a Secret
- Blog deployment uses SHA-pinned image tags (`ghcr.io/derio-net/blog:<sha>`) â€” CI commits the tag update to `deployment.yaml`, ArgoCD syncs automatically. No manual `rollout restart` needed
- Falco: installing a binary at container runtime (`apk add` / `pip install` / `npm i -g`, etc.) writes it to the container's writable upper layer and trips the **Critical** rule "Drop and execute new binary in container" (`EXE_UPPER_LAYER`, MITRE TA0003) â€” which pages via Telegram. This is a *benign-true-positive*, not a Falco bug: bake the tool into a digest-pinned image (read-only base layer) rather than muting the rule. First hit: headscale-backup CronJob's `apk add sqlite` at 03:00 â†’ switched to `alpine/sqlite` pinned by digest.
- Falco priority routing on Hop: floor is `notice`; only `Critical` reaches Telegram, everything `â‰Ą notice` goes to VictoriaLogs (`192.168.55.225:9428`) via the Loki push protocol. To audit what's actually firing without paging: `priority:<level> | stats by (rule) count()` in LogsQL. "Contact K8S API Server From Container" (Notice, mostly ArgoCD reconcile) is the expected high-volume benign baseline.
- CrowdSec bans nothing if the LAPI's `data`/`config` volumes are emptyDir: every LAPI restart wipes the SQLite DB incl. the agent's registered-machine row, so the agent crashloops with `ent: machine not found` (`cscli machines list` empty), parses zero Caddy logs, and produces zero decisions â€” ArgoCD still shows Synced/Healthy (silent). Fixed (2026-06-19) by persisting both LAPI volumes onto static hostPath PVs (`type: DirectoryOrCreate`, subdirs of `/var/mnt/hop-data/crowdsec/` on the already-attached Hetzner Volume â€” no new volume), bound to the chart PVCs (`crowdsec-{db,config}-pvc`) from the PV side via `claimRef` since the chart's PVC template has no `volumeName`. `storageClassName: hetzner-volume` is MANDATORY in values â€” Hop has no default StorageClass, so an unset class hangs the PVC `Pending` forever. The `claimRef.name` MUST track the chart's `<releaseName>-db-pvc` naming (guarded by `scripts/tests/test_crowdsec_lapi_persistence.py` â€” a LOCAL guard; no CI runs `scripts/tests/`). Persisting `config` also makes the community-blocklist console enrollment durable across restarts. Two follow-on traps: (1) the agent is a DaemonSet that registers its machine ONLY in an initContainer, so the emptyDirâ†’persistent cutover does NOT self-heal â€” `kubectl -n crowdsec-system rollout restart daemonset/crowdsec-agent` once to re-register against the fresh DB; (2) `hostPath`+`DirectoryOrCreate` is single-node-safe ONLY â€” `nodeAffinity: hostname Exists` is a tautology, so a 2nd Hop node would silently get a fresh empty dir and lose the DB; pin to `hop-1` before scaling out.
- CrowdSec STILL bans nothing even with a healthy agent if `container_runtime` is wrong: Talos = **containerd** (CRI logs `<ts> stdout F {json}`), but the chart defaults `container_runtime: docker` â†’ acquisition labels logs `type: docker` â†’ `docker-logs` parser does `JsonExtract(Raw,"log")` which is EMPTY for a CRI line â†’ `caddy-logs`' `UnmarshalJSON` fails â†’ access logs unparsed â†’ zero scenarios â†’ zero bans (ArgoCD green, agent Running). Diagnose with `cscli metrics`: Acquisition shows `Lines read > 0` but `Lines parsed = 0`, and Parser metrics show ONLY `docker-logs`, no `caddy-logs`. Fix (2026-06-19): set top-level `container_runtime: containerd` in `clusters/hop/apps/crowdsec/values.yaml` â†’ `cri-logs` strips the envelope; the acquisition's `program: caddy` label still reaches `caddy-logs`. Note `cri-logs` filter is `type == 'containerd'` (NOT `'cri'`) and does NOT set `evt.Parsed.program` â€” so the `program: caddy` acquisition label is load-bearing (both halves guarded by `scripts/tests/test_crowdsec_cri_logs.py`). Verify end-to-end with `cscli explain --type containerd --labels program:caddy -f <caddy container log>`.
- CrowdSec STILL goes blind (third silent failure) on **Kubernetes log rotation** even with a healthy agent + correct runtime: it tails `/var/log/containers/caddy-*_caddy-system_*.log`, which are **symlinks** to the kubelet's `/var/log/pods/.../0.log`. The chart hardcodes `force_inotify: true` and defaults `poll_without_inotify: false`, so the per-file tailer follows the resolved inode via inotify and CANNOT follow a kubelet **size-rotation** (`containerLogMaxSize`, renames `0.log`â†’`0.log.<ts>` WITHOUT a pod restart) â€” it logs `Re-opening moved/deleted file` then hangs forever in `Waiting for ...log to appear`, parsing zero lines (agent `Running 1/1`, ArgoCD green, zero bans; bit ~19h on 2026-06-20, last ban 09:55Z). The Caddy pod uptime is the tell: pod NOT restarted (e.g. 27d) but acquisition frozen â†’ it was a size-rotation, not pod churn. Diagnose with `cscli metrics`: the source's `Lines read`/`parsed` are non-zero but FROZEN. The agent prints the exact fix at startup (`...is a symlink, but inotify polling is enabled...Consider setting poll_without_inotify to true`). Fix (2026-06-21): add `poll_without_inotify: true` to the `agent.acquisition[]` entry in `clusters/hop/apps/crowdsec/values.yaml` (the ONLY chart-overridable lever â€” `force_inotify` is hardcoded but orthogonal: it governs directory-watch for NEW files, not the per-file tailer). Polling is stat-based â†’ detects rotation/truncation â†’ re-attaches. Guarded by `scripts/tests/test_crowdsec_log_rotation_poll.py`. Deploying it rolls the DaemonSet, which resumes tailing.
